# 講師專用 — 解答大綱與評分對照

**閱讀對象：** 講師與助教。**不建議**原封不動發給學員當「標準答案」；技術 Lab（尤其滲透敘述）應以**驗收產物 + 口試**為主，避免鼓勵背答案或越權複製 exploit。

**搭配文件：** 各課 [`labs/labNN/cursor_prompts.md`](../labs/README.md)、[`INSTRUCTOR_PROMPT_GUIDE.md`](INSTRUCTOR_PROMPT_GUIDE.md)、[`TRACK_C_PENTEST.md`](TRACK_C_PENTEST.md)。

---

## 評分原則（全課適用）

| 維度 | 及格（Pass） | 優良（Distinction） |
|------|----------------|----------------------|
| **範圍** | 產物與講義白名單一致，無「順手多掃」敘述 | 主動標註邊界與假設，能說明為何不敢越界 |
| **可稽核** | log／截圖／檔案路徑能對上文字陳述 | 時間線清楚，指令 argv 可追溯 |
| **治理** | RoE／核准 gate 有遵守（Lab08–11） | 能對照真實專案中誰該簽字、如何存證 |
| **AI 使用** | 能說明何時用 AI、何時必須人工 | 能指出 AI 幻覺或過度侵略建議並糾正 |

**Track A 客觀對照：** `pytest` 通過與 schema／CLI 行為；講師仍可要求口試解釋設計取捨。

## Track A — Lab00–05

### Lab00 — Repo 理解

- **預期產物：** 目錄結構說明（含 `hexstrike_lab/`、`configs/`、`tests/`）、從 CLI 到輸出的資料流（文字或圖）。
- **及格：** 能對應至少 3 個頂層目錄職責；流程不與「直接改 JSON 手刻報告」混淆。
- **常見不合格：** 只貼 README 首段、無執行路徑概念。
- **口試：** `run` vs `assess` vs `pipeline` 差一個維度是什麼？

### Lab01 — Adapter

- **預期產物：** 自訂或修改 `ToolAdapter` 概念實作：安全執行（避免 `shell=True`）、JSON 輸出、錯誤處理；normalize 範例。
- **及格：** 介面與 `run_adapter` 契約一致；失敗時有結構化錯誤而非裸 traceback 當唯一輸出。
- **優良：** 明確 `validate_target` 或等效白名單思維。
- **口試：** 為何列表參數比字串 shell 安全？

### Lab02 — Orchestrator

- **預期產物：** 編排邏輯說明 + 與 profile／步驟條件相關的設計。
- **及格：** 能描述「何時跑下一步」的規則，而非單一腳本順序死寫。
- **口試：** `run_when` 與 LLM 自由選工具的根本差異？

### Lab03 — Pipeline

- **預期產物：** 端到端 pipeline 敘述 + 從 JSON 產 Markdown 的設計或實作片段。
- **及格：** 產物路徑／階段與本 repo `pipeline` 概念對齊。
- **口試：** dry-run 為何仍有教學價值？（可銜接 Lab14）

### Lab04 — RFP → 掃描

- **預期產物：** `rfp_scan_requirements` 與 profile 的對應說明；可追溯需求 ID。
- **及格：** 至少一條需求對到工具／產物欄位。
- **口試：** 若客戶改 RFP 一句話，哪裡要改設定？

### Lab05 — 滲測工作流與證據

- **預期產物：** 多步工作流 + 證據 schema 設計或欄位對照。
- **及格：** 與 `pentest_workflow.yaml`／`evidence_schema` 意涵一致，非空談。
- **口試：** `evidence_record` 最少要留住哪幾類資訊給稽核？

## Track B — Lab06–15

### Lab06 — MCP 與後端

- **預期產物：** 連線示意圖、health 成功證明、Ollama／`llama3` 依講義要求之 list／推論節錄、元件責任說明。
- **及格：** MCP → HTTP server →（選）Ollama 層次正確；路徑／埠可驗證。
- **常見不合格：** 只有截圖無法對應實際 IP／埠；把 `hexstrike_lab` CLI 誤稱為 MCP。
- **口試：** stdio MCP 與 HTTP server 各解決什麼問題？

### Lab07 — 結構化偵察

- **預期產物：** 掃描摘要表、**至少一條完整 argv 溯源**、簡答（LLM vs `run_when`）。
- **及格：** 表內目標落在白名單；溯源來自 log／工具輸出而非純模型口述。
- **Track C：** 有策略段落 + 人工刪減證明者加分。
- **口試：** 若模型宣稱掃了某 IP，你如何驗證真假？

### Lab08 — JSON 與驗證節制

- **預期產物：** JSON（或節錄）、驗證腳本、執行輸出、**核准紀錄**、evidence 心智表、200 字真偽陽性分析。
- **及格：** **無核准紀錄則驗證腳本執行視同不合格**；腳本僅連單一核准 URL。
- **優良：** 能具體寫出誤報來源與降級驗證方式。
- **不提供：** 完整可執行 exploit／大量請求腳本「範本」；講師自行保管課堂用參考碼即可。

### Lab09 — DVWA／授權 Web

- **預期產物：** URL 聲明、sqlmap 或手動 log、變形 payload 列表、倫理短文；Track C 另要 raw request 節錄。
- **及格：** URL 與講義一致；有失敗嘗試紀錄；無越權目標。
- **常見不合格：** 只有成功截圖、無 log；payload 適用於任意站點。
- **口試：** Bug Bounty 下為何不能讓 AI 自動開 sqlmap？

### Lab10 — 程式修補與 JSON

- **預期產物：** Python 3 可執行碼、JSON 範例、（選）listener 證據；bypass 敘述（Track C）。
- **及格：** 無 `eval`／危險 `os.system` 未審查即交；JSON 符合單一物件約定。
- **反彈 shell：** 僅在講師核准與隔離網；預設評分以程式與審查過程為主。
- **口試：** AI 改 exploit 為何不能完全信任？

### Lab11 — 綜合與報告

- **預期產物：** Markdown 報告、`pipeline` 產物路徑證明、半頁對照、Track C 三格 attack-chain 表（及選修 ATT&CK／flag）。
- **及格：** 報告章節齊備且**每節有對應證據**；對照文誠實比較機械 findings vs 敘事。
- **常見不合格：** 報告像泛用模板、無標的專屬 log。
- **口試：** 客戶只要合規證據時你交哪一份？

### Lab12 — RoE

- **預期產物：** 完整 RoE（含範圍 CIDR／URL、禁止項、聯絡與中止）、互評三問、一句誤用情境。
- **及格：** 禁止項與白名單非空；模糊地帶有被問出來。
- **優良：** 明確寫出人審觸發點（與 MCP 操作掛鉤）。
- **口試：** AI 能否代替授權人簽字？

### Lab13 — 證據對照

- **預期產物：** 對照表 + 200 字補欄位說明。
- **及格：** 同一事件兩種表述欄位大致對齊 schema；指出 MCP 易缺欄位。
- **口試：** 若 command argv 遺失，稽核上會發生什麼事？

### Lab14 — CI gate

- **預期產物：** workflow 或等同設定、green run 截圖、100 字 dry-run 意義。
- **及格：** CI 內**未**對真實目標 `--execute`；有 schema 或 validate 步驟。
- **口試：** `run_id` 變動時如何穩定驗證路徑？

### Lab15 — 選修

- **預期產物：** 依所選 15A–15D 之講義要求；15D 需不越權、可稽核。
- **及格：** 與選修主題一致且有 RoE／範圍自省。
- **口試：** 若工具清單與 hexstrike-ai 不一致如何裁切？
